The Key Insight

The laptop does all the heavy lifting (YOLO inference, PID math). The ESP32 is just a fast, reliable motor driver that listens for simple MOTOR,left,right UDP packets and executes them immediately. This split keeps the embedded code simple and the intelligence in Python.

02 / Hardware

What's Inside the Robot

The hardware stack is minimal and deliberately chosen — powerful enough to run real-time control at 30Hz, simple enough to wire in an afternoon.

Component Part Role in the System
Microcontroller ESP32 WROVER WiFi + Bluetooth, receives UDP motor commands, drives PWM signals. The WROVER variant has extra PSRAM useful for larger buffers.
Motor Driver TB6612FNG Dual H-bridge driver — takes two PWM + direction signals and drives both motors independently. Handles up to 1.2A continuous per channel. STBY pin must be pulled HIGH to enable.
Motors (×2) DG01D Geared DC motors with encoder outputs. Used in differential-drive configuration — vary left vs right speed to steer. The encoder ticks are logged as telemetry back to the laptop.
Power 9V Battery Powers motors via TB6612FNG. The ESP32 should be powered separately (3.3V regulated) to avoid noise from the motors browning out the MCU.
Camera Smartphone Streams MJPEG video at 8080/video using the IP Webcam app (Android) or similar. The Python code reads this as a chunked HTTP stream and decodes JPEG frames.
Vision PC Laptop / Desktop Runs YOLO inference and the PID loop. Can use CUDA (GPU) for faster inference — falls back to CPU automatically.

Wiring the TB6612FNG to ESP32

Pin Mapping (ESP32 WROVER)
// ── RIGHT Motor (Motor A) ─────────────────────────── MOTOR_A_PWM → GPIO 13 // PWM speed signal MOTOR_A_IN1 → GPIO 15 // Direction bit 1 MOTOR_A_IN2 → GPIO 18 // Direction bit 2 // ── LEFT Motor (Motor B) ──────────────────────────── MOTOR_B_PWM → GPIO 19 // PWM speed signal MOTOR_B_IN1 → GPIO 32 // Direction bit 1 MOTOR_B_IN2 → GPIO 33 // Direction bit 2 MOTOR_STBY → GPIO 14 // Pull HIGH to enable driver // ── Direction Logic ───────────────────────────────── // Forward: IN1=HIGH, IN2=LOW // Backward: IN1=LOW, IN2=HIGH // Brake: IN1=LOW, IN2=LOW
⚠️
Common Wiring Mistake

If your robot spins in place instead of going straight, the left and right motors are likely wired in opposite polarities. In the code, the commander sends -left_speed, right_speed — this inversion corrects for one motor being physically mounted in reverse. Adjust the sign if your robot behaves unexpectedly.

03 / Training Data

From Raw Images to a YOLO Model with Roboflow

Before the robot can "see" a ball, you need to teach it what a ball looks like. That means collecting images, labeling bounding boxes, and training a model. Roboflow makes this entire pipeline visual and beginner-friendly.

Step 1 — Collect Images

Variety is everything. Capture your target object under different conditions — different lighting, backgrounds, angles, distances, and partial occlusions. A ball rolling on carpet, on tile, near walls, in shadow, partially behind furniture. Aim for at least 200–300 images for a simple single-class detector. More is always better.

📸
Quick Tip: Use Your Robot's Camera

Collect images from the exact same phone and height the robot will use during inference. A model trained on 4K DSLR photos will perform poorly on a wobbly 720p phone stream mounted 20cm off the ground. Match your training domain to your deployment domain.

Step 2 — Label on Roboflow

Upload your images to roboflow.com, create a new project (Object Detection), and draw bounding boxes around your target in each image. Label the class — e.g. ball. Roboflow's labeling tool is drag-and-drop; it also has an AI-assisted auto-label feature that can pre-fill boxes for you to correct.

Step 3 — Augmentation & Export

Roboflow applies augmentations automatically — flips, rotations, brightness changes, blur — effectively multiplying your dataset 3–10× for free. Then export in YOLOv8 format.

train_model.py — Training YOLOv8 on your Roboflow dataset
from ultralytics import YOLO from roboflow import Roboflow # Step 1: Download your Roboflow dataset rf = Roboflow(api_key="YOUR_API_KEY") project = rf.workspace("your-workspace").project("ball-detector") dataset = project.version(1).download("yolov8") # Step 2: Load a pretrained YOLOv8 nano model (fast, lightweight) # Options: yolov8n (nano), yolov8s (small), yolov8m (medium) model = YOLO("yolov8n.pt") # Step 3: Fine-tune on your dataset results = model.train( data=f"{dataset.location}/data.yaml", epochs=50, # number of training passes imgsz=320, # smaller = faster inference on laptop batch=16, # images per training step name="ball_detector" ) # Step 4: Your trained model is saved at runs/detect/ball_detector/weights/best.pt # Copy best.pt to your project folder and set model_path = 'best.pt' print("Training complete! Model saved to runs/detect/ball_detector/weights/")
🎯
What's imgsz=320 doing?

YOLOv8 resizes every input frame to a square before inference. 320×320 is much faster than 640×640 (4× fewer pixels) with only a small accuracy loss. For a ball-tracking task where the object is large relative to the frame, 320 is plenty. If you're detecting small distant objects, increase to 640.

04 / Control System

Vision Pipeline & PID Control

Once YOLO finds the ball, Python computes two error signals and feeds them into two independent PID controllers — one for steering, one for distance.

The Two Error Signals

ANGLE ERROR

How far left/right is the ball from the center of the frame?

angle_error = (cx - width/2) / width

Ranges from -0.5 (far left) to +0.5 (far right). Zero means ball is perfectly centered.

DISTANCE ERROR

How close/far is the ball from the target size?

dist_error = (area - TARGET_AREA) / TARGET_AREA

Positive = ball too close (too large). Negative = too far. Target area is ~143,000 px².

PID Controller Explained

A PID controller is a feedback loop with three terms. Each term "feels" the error differently and contributes to the output command:

P
Proportional
Kp × error
Kp=10
+
I
Integral
Ki × ∫error dt
Ki=0 (off)
+
D
Derivative
Kd × Δerror/dt
Kd=3
→
Output
Turn or Speed
command
📊
P — Reacts to current error

Bigger error = bigger correction. Too high → robot overshoots and oscillates. Too low → lazy, slow tracking.

📉
D — Reacts to rate of change

Slows the robot down as it approaches the target (like damping). Prevents overshoot. Set too high → system becomes sluggish.

⚖️
I — Reacts to accumulated error

Corrects for steady-state bias (e.g. one motor slightly weaker). Currently set to 0 — can be activated if the robot drifts over time.

Mixing Steering + Speed

tracker.py — How turn and speed are mixed into left/right motor commands
# turn_cmd: positive = turn right, negative = turn left # base_speed: constant forward drive while tracking base_speed = 15 # constant crawl-forward while tracking turn_cmd = pid_turn.update(angle_error) # Differential drive mixing: # Left wheel faster → robot turns right (and vice versa) tgt_left = base_speed + turn_cmd tgt_right = base_speed - turn_cmd # Example: ball is to the right (angle_error = +0.2) # turn_cmd ≈ +10 → tgt_left=25, tgt_right=5 → turns right ✓ # Cap at MAX_SPEED while preserving ratio max_raw = max(abs(tgt_left), abs(tgt_right)) if max_raw > MAX_SPEED: scale = MAX_SPEED / max_raw tgt_left *= scale tgt_right *= scale # Speed ramping — don't jump instantly to target speed # This prevents wheel slip and sudden jerks cur_left = ramp(cur_left, tgt_left, MAX_ACCEL) cur_right = ramp(cur_right, tgt_right, MAX_ACCEL)

State Machine

The robot isn't always in the same mode. A 4-state machine handles the transition between finding the ball, confirming it, tracking it, and searching when it disappears:

⬛ Stopped

Ball not seen for > 1 second. Motors off. PIDs reset. Robot waits until something is detected.

🟠 Searching

Ball was just lost (<1s ago). Robot slowly spins in the last known direction of the ball to re-acquire.

🟡 Acquiring

Ball just appeared. Speed limited to 40–100% over 5 frames to avoid chasing false positives.

🟢 Tracking

Ball confirmed. Full-speed PID control active. Both turn and distance PIDs running simultaneously.

05 / Embedded Firmware

ESP32 Motor Controller

The ESP32 firmware is intentionally minimal — connect to WiFi, listen on UDP port 5001, parse MOTOR,L,R packets, and drive the motors. That's it.

Communication Protocol

COMMAND FORMAT

MOTOR,L,R

e.g. MOTOR,-20,25
Values: -255 to 255

STOP COMMAND

STOP,OFF

Sent at startup and shutdown to zero motors

TELEMETRY BACK

ENC,L,R,CL,CR

Encoder ticks + commanded speeds sent back to laptop on port 5002

PWM Motor Control

The ESP32's ledcWrite generates a PWM signal that controls motor speed. The TB6612FNG H-bridge interprets IN1/IN2 pin states to set direction, and the PWM duty cycle sets speed. At 0% duty = stop, 100% duty = full speed.

motor_control.ino — driveMotor function logic
void driveMotorA(int speed) { // RIGHT motor if (speed > 0) { digitalWrite(MOTOR_A_IN1, HIGH); // Forward digitalWrite(MOTOR_A_IN2, LOW); } else if (speed < 0) { digitalWrite(MOTOR_A_IN1, LOW); // Reverse digitalWrite(MOTOR_A_IN2, HIGH); } else { digitalWrite(MOTOR_A_IN1, LOW); // Brake (coast) digitalWrite(MOTOR_A_IN2, LOW); } ledcWrite(PWM_CHANNEL_A, abs(speed)); // 0–255 duty cycle } // UDP loop — non-blocking, checks for packets every loop iteration void loop() { int packetSize = udp.parsePacket(); if (packetSize > 0) { char buf[128]; int len = udp.read(buf, sizeof(buf) - 1); buf[len] = '\0'; handleCommand(String(buf)); // Parse and execute } // No delay() — stays responsive at full loop speed (~1 MHz) }
🔌
Why UDP instead of TCP?

UDP is connectionless and has no retransmission. For motor control, a stale motor command delivered late is worse than a dropped one — the robot would lurch. UDP drops packets gracefully. The Python loop sends a fresh command every 30ms anyway, so missing one rarely matters.

Tuning the PID — Where to Start

TURN PID

If robot oscillates wildly when centering: lower Kp or raise Kd.

Current: Kp=10, Ki=0, Kd=3

DISTANCE PID

If robot rushes past the target then reverses: lower Kp and raise Kd.

Current: Kp=10, Ki=0, Kd=5

TARGET AREA

Set TARGET_AREA to the pixel area of the ball at your desired follow distance.

Print debug['area'] while the ball is at target distance, then use that value.

Where to go from here

Multi-object tracking — Change TARGET_OBJECT to any COCO class name (person, bottle, dog) and it works immediately with the base YOLOv8 weights, no Roboflow needed.

On-robot inference — Deploy the YOLO model to a Jetson Nano or Raspberry Pi 5 mounted on the robot itself, cutting the WiFi latency entirely.

Encoder-based control — The telemetry system already reads encoder ticks. Use those to build a closed-loop wheel speed controller, so both motors maintain equal speeds regardless of load.
